# ClipSoon 架构设计

## 1. 设计原则

- 本地优先：核心功能不依赖网络；SQLite 和图片文件是唯一持久数据。
- 平台隔离：全局按键、前台窗口恢复和粘贴发送都在平台适配层。
- 可测试：时钟、历史仓库、设置、快捷键输入、剪贴板和目标窗口均可替换。
- 故障降级：即使无法恢复目标窗口，选中的内容也已写入系统剪贴板。
- 面板语义：主窗口是临时命令面板，不是传统常驻主窗口。

## 2. 模块关系

```text
Qt/Native clipboard ──> ClipboardController ──> HistoryRepository(SQLite + PNG)
                                              │
global keys ──> HotkeyStateMachine ──> App ──> MainWindow ──> SearchEngine
                                              │
                                   SelectionSender
                                      │       │
                               Qt clipboard  ForegroundTarget + synthetic paste

Settings ──> LaunchAtLoginManager ──> LaunchAgent (macOS) / HKCU Run (Windows)
```

生产代码有五个主体文件：`core.py` 放模型、设置和 SQLite；`search.py` 单独放顺序匹配、评分
公式和内存索引，便于定位与增强；`system.py` 放剪贴板、快捷键和平台边界；`ui.py` 放全部
Qt 界面；`app.py` 只做依赖装配。没有按“控制器/服务/仓库”继续拆目录，保持效率工具代码
短路径、易定位。

## 3. 数据与去重

文本哈希基于 UTF-8 原文；文件哈希基于规范化绝对路径序列；图片先转换为无损 PNG 后对字节
哈希。数据库对 `content_hash` 建唯一索引。重复采集在一个事务中更新 `updated_at`、来源和
使用时间，不新增记录。

图片存入应用数据目录 `images/<sha256>.png`，数据库只记录相对文件名。删除历史后只有在
没有其他记录引用时才删除图片，避免数据库与磁盘状态分裂。

## 4. 搜索排序

检索先做 Unicode NFKC 和 `casefold` 归一化。空查询直接按 `updated_at` 倒序。有查询时，
查询的每个字符必须按原顺序出现在同一个检索字段中，否则不匹配；完全相同固定为最高分
`1,000,000`。其余得分固定为：

```text
score = 5000 × (查询长度 / 条目长度)
      + 3500 × (相邻命中对数 / 查询相邻对数)
      + 1500 × (查询长度 / 匹配路径跨度)
```

这样同为连续命中时条目越短分越高，同长度时命中越连续分越高。搜索得分不混入新近度、
频率或置顶加成；只在得分完全相同时用 `updated_at` 决胜。连续子串和不可能匹配走原生
`find` 快路径；其余情况用稀疏动态规划精确选择综合得分最高的有序匹配路径，而不是用可能
低估连续性的贪心近似。

## 5. 平台策略

全局按键由 `pynput` 产生统一的按下/松开事件，纯状态机识别双击修饰键或组合键。macOS
后台使用 220 ms 定时器只读取 `NSPasteboard.changeCount`，变化时再读取 Qt MIME，避免隐藏时
漏采；Windows 使用 Qt 原生剪贴板事件并以系统 sequence number 去重。窗口弹出前，macOS
记录前台进程 PID，Windows 记录前台窗口句柄。发送时先写剪贴板并隐藏面板，等待可配置延迟
后恢复目标并发送 `Cmd+V` 或 `Ctrl+V`。

用户级开机自启动由 `LaunchAtLoginManager` 隔离：macOS 原子写入
`~/Library/LaunchAgents/com.clipsoon.app.plist`，Windows 写入
`HKCU\Software\Microsoft\Windows\CurrentVersion\Run` 的 `ClipSoon` 字符串值。PyInstaller 环境只记录
当前应用可执行文件；源码环境记录当前虚拟环境解释器和 `-m clipsoon`，Windows 优先使用同目录
`pythonw.exe` 避免登录时出现控制台。设置保存失败时保留原设置值并反馈；已开启时每次应用启动
都会重写启动项，使应用升级或移动后的可执行路径得到刷新。

## 6. 性能路径

应用启动后常驻并预构建窗口、内存搜索索引和列表 delegate。热键呼出只记录前台目标、计算
当前屏幕位置并显示已经存在的窗口，不查数据库。搜索字段已归一化，列表由 `QListView` 虚拟
绘制；图片 PNG 编码、数据库写入、列表缩略图和右侧大图解码均放入 Qt 线程池。选中路径只同步
更新当前索引、背景和元信息，图片先显示加载占位，后台返回缩放后的 `QImage` 后才在 GUI 线程
转换为 `QPixmap`，并按路径和目标尺寸缓存。日志记录冷启动、呼出和超过 20 ms 的搜索，用于
回归定位。

可选状态记忆只保存在当前进程内：面板真正隐藏前以单调时钟记录类型 Tab、搜索词、稳定条目 ID
集合和当前焦点 ID，并启动单次到期计时器。时限内重显时先恢复类型、搜索词和过滤结果，再按
ID 恢复选择；到期时主动丢弃整个快照、恢复“全部”、清空搜索并把隐藏面板重置到完整列表第一项。持久化配置仍沿用
`remember_selection` 键以兼容 0.9.1 用户设置，但该开关的产品语义已扩展为完整面板状态。

主面板位置以 `panel_x`、`panel_y` 两个可选逻辑坐标写入同一 JSON 设置。首次运行仍按当前鼠标
屏幕居中偏上显示；用户只能从非交互空白区域开始拖动，释放鼠标且位置确实改变后才由面板信号
触发一次设置落盘。后续显示优先选择保存坐标所在屏幕，并按该屏幕可用区域约束完整窗口；旧屏幕
不存在时回退到当前鼠标屏幕，避免窗口留在不可见区域。

开发验收统一通过仓库根目录的 `run.command` 或 `run.bat` 从当前源码启动。启动
脚本先终止本仓库的旧打包/源码实例，避免单实例锁让旧 `dist` 产物继续驻留并造成源码与界面
不一致。发布和最终包验收分别使用根目录的 `build_macos.command` 或 `build_windows.bat`；它们
转发到 `scripts/` 下的平台 PyInstaller 实现。

GitHub Actions 的 `.github/workflows/release.yml` 在推送 `v*` 标签时运行。Windows 任务固定使用 x64
托管 runner，macOS 任务固定使用 macOS 15 Apple Silicon runner；两者分别生成 ZIP 后作为独立
artifact 上传。发布任务只在两个平台构建均成功后运行，下载产物、生成 SHA-256 校验文件，
再使用最小的 `contents: write` 权限创建或更新对应的 GitHub Release。

界面主题由 `_style_sheet()` 提供主窗口与设置窗口的基础颜色，但弹出层不能只依赖父窗口继承。
macOS 会把 `QComboBox` 的下拉视图放在独立顶层窗口中，应用主题与系统主题不一致时可能继续使用
系统背景；因此两个设置下拉框会直接为弹出视图设置 Base/Text/Highlight 调色板及普通、悬停、
选中、禁用 QSS。历史右键菜单也接收 ClipSoon 当前主题值，而不是再次从系统调色板推断深浅；
系统托盘菜单保持操作系统原生外观。


操作系统可能拒绝跨进程激活；此时不能把“已复制”误报成“已粘贴”。应用保留剪贴板内容并
给出降级提示。

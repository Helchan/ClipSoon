# ClipSoon 架构设计

## 1. 设计原则

- 本地优先：核心功能不依赖网络；SQLite 和图片文件是唯一持久数据。
- 平台隔离：全局按键、前台窗口恢复和粘贴发送都在平台适配层。
- 可测试：时钟、历史仓库、设置、快捷键输入、剪贴板和目标窗口均可替换。
- 故障降级：即使无法恢复目标窗口，选中的内容也已写入系统剪贴板。
- 面板语义：主窗口是临时命令面板，不是传统常驻主窗口。

## 2. 模块关系

```text
macOS pasteboard / pynput ───────────────┐
                                        v
Windows hotkey host ── JSON pipe ──> supervisor ──> App ──> MainWindow ──> SearchEngine
Windows clipboard host ─ manifest ─> ClipboardController ──> HistoryRepository(SQLite + PNG)
                                                    │
                                             SelectionSender
                                                    │
                     native clipboard broker <─ ACK/verify ─> ForegroundTarget + synthetic paste

Settings ──> LaunchAtLoginManager ──> LaunchAgent (macOS) / HKCU Run (Windows)
```

`core.py` 放模型、设置和 SQLite；`search.py` 放检索；`ui.py` 放 Qt 界面；`system.py` 放主进程中的
平台适配和发送链路；`app.py` 只做依赖装配。`launcher.py` 是不导入 Qt 的最薄进程入口，先识别
`--windows-helper`，普通模式才加载 UI。Windows 另有 `windows_hotkey_host.py`、
`windows_clipboard_host.py` 两个原生宿主，以及负责启动、协议校验、心跳和重建的
`windows_workers.py`。因此冻结包仍只有一个 `ClipSoon.exe`，但运行时是一个 UI 主进程加两个隔离
子进程；子进程在单实例锁和 QApplication 之前分派，不会递归启动完整应用。

## 3. 数据与去重

文本哈希基于 UTF-8 原文；文件哈希基于规范化绝对路径序列；图片先转换为无损 PNG 后对字节
哈希。数据库对 `content_hash` 建唯一索引。重复采集在一个事务中更新 `updated_at`、来源和
使用时间，不新增记录。

图片存入应用数据目录 `images/<sha256>.png`，数据库只记录相对文件名。删除历史后只有在
没有其他记录引用时才删除图片，避免数据库与磁盘状态分裂。

文件记录只保存规范化绝对路径，不复制源文件。启动后立即、隐藏驻留期间每 3 秒以及每次呼出
面板时，单个后台守护工作线程会异步扫描文件记录；多文件记录中任一路径明确抛出 `FileNotFoundError` 或
`NotADirectoryError` 后，还会先确认对应卷、UNC 共享或映射盘根目录可访问，再删除整条记录。
权限拒绝、断开的共享、拔出的移动盘等不确定状态保留记录，避免把临时故障误判为永久删除。
每条记录持久保存单调递增的 `revision`；扫描在数据库锁外访问
文件系统，回库后只按 `(id, kind, revision)` 比较并删除，扫描期间重新复制、使用或置顶的记录不会
被旧结果误删。发送当前文件项前由最多两个后台守护工作线程重新解析仓库中的最新 revision，GUI 线程不执行可能阻塞的
网络路径 `stat`；失效项不会写入伪造的 `CF_HDROP`，而是删除并刷新列表。

## 4. 搜索排序

检索先做 Unicode NFKC 和 `casefold` 归一化。空查询直接按 `updated_at` 倒序。有查询时，
查询的每个字符必须按原顺序出现在同一个检索字段中，否则不匹配；完全相同固定为最高分
`1,000,000`。其余得分固定为：

```text
score = 5000 × (查询长度 / 条目长度)
      + 3500 × (相邻命中对数 / 查询相邻对数)
      + 1500 × (查询长度 / 匹配路径跨度)
```

这样同为连续命中时条目越短分越高，同长度时命中越连续分越高。搜索得分不混入新近度、
频率或置顶加成；只在得分完全相同时用 `updated_at` 决胜。连续子串和不可能匹配走原生
`find` 快路径；其余情况用稀疏动态规划精确选择综合得分最高的有序匹配路径，而不是用可能
低估连续性的贪心近似。

## 5. 平台策略

macOS 全局按键继续由 `pynput` 送入纯状态机；状态机按每个按键的按下时间清理漏失的松开事件。
Windows 不观察前台应用的键盘输入，也不使用 `pynput`、Raw Input、低级键盘 hook 或
`GetAsyncKeyState` 轮询。所有 Windows 呼出快捷键都交给
`RegisterHotKey + MOD_NOREPEAT`，由操作系统向独立宿主的消息队列投递 `WM_HOTKEY`。Windows
默认组合键为 `Ctrl+Shift+Space`；旧版持久化的双修饰键或 Win32 解析器不支持的配置在应用装配时迁移并保存，设置页在
Windows 上只提供可验证的组合键录制。当前会话唯一 mutex 避免正式版与调试实例同时响应。每次触发
时宿主先快照 `GetForegroundWindow`，再调用
`AllowSetForegroundWindow(parent_pid)` 尝试把本次输入授予的前台切换资格交给 UI，最后在
`hotkey` 协议事件中携带目标 HWND 和授权结果；UI 不再在 IPC 延迟后重新猜测原目标。注册失败是
明确的配置错误；注册成功后由 Windows 持有到宿主线程退出，不存在本地 handle 仍显示健康、系统却
静默移除监听器的分裂状态。设置切换组合键时保留最近一次收到 `ready` 的配置；候选组合键若因占用
注册失败，主进程在下一轮 Qt 事件中回写旧配置并重启旧注册，避免一次无效保存永久丢失呼出入口。
若应用启动时自定义组合键冲突且还没有收到过任何配置的 `ready`，则改用并持久化默认组合键；
默认组合键也冲突时停止回退，只保留托盘入口并明确报告，避免反复重启。
宿主每 500 ms 发心跳；异常退出时由 supervisor 重建并重新注册。主进程
执行健康判断前先消费子进程已经缓冲的输出，避免 GUI 短暂忙碌后误杀仍持续发心跳的宿主。空闲阶段
1.75 秒无活动或 4 秒无
启动握手时终止并重建；原生剪贴板读取使用独立的 10 秒硬超时，避免把合法的大数据内存复制按普通
心跳误杀。每代进程使用父进程生成的 nonce、严格递增事件号和 PID 握手，旧进程迟到消息不会进入
新代次。控制管道断开或父进程死亡时宿主直接退出，即使自己的 Win32 消息线程已经卡住也不会成为
孤儿进程。应用崩溃后若旧宿主尚未来得及释放会话 mutex，新代 supervisor 会按退避策略继续尝试，
不会把这次短暂占用误判为永久禁用；真正的 `RegisterHotKey` 组合键冲突才停止当前配置的重试。

macOS 后台使用 220 ms 定时器只读取 `NSPasteboard.changeCount`，变化时再读取 Qt MIME。Windows
主进程不再读取 `QClipboard.mimeData()`：剪贴板宿主使用 `AddClipboardFormatListener` 和
`GetClipboardSequenceNumber` 合并变化，通知后先等待 70 ms，再以 80/160/320/640/1000 ms 退避；
每次 `OpenClipboard` 都在 `finally` 中配对 `CloseClipboard`。文本、HDROP 文件、原生 PNG 与
CF_DIB/CF_DIBV5 在宿主中复制到私有 IPC 目录，短 JSON 管道只传原子 manifest 文件名。关闭系统剪贴板
后进入 materializing 阶段，由独立线程继续发心跳，避免 256 MiB 图片转换、写入和 `fsync` 被 1.75 秒
阈值误杀；该阶段仍有 120 秒硬上限。主进程后台校验目录、大小、协议、序号和图片尺寸后入库并删除
临时文件。所有 IPC 文件带 helper session 前缀，新代启动前会保留仍在消费的 session 并回收其他代次
遗留。若 `GetClipboardData` 因延迟渲染永久阻塞，只有剪贴板宿主被替换，Qt GUI 与热键宿主不受影响；
重建时从“最后成功消费序号”重放当前剪贴板。清空、暂停或写回时递增采集代次，已排队的旧 manifest
只能被清理，不能复活历史。

Windows 剪贴板宿主同时承担采集和发送，是系统剪贴板的唯一所有者；GUI 进程不调用
`QClipboard.mimeData()`，也不执行 Qt/OLE 写入。发送内容先在后台生成带 session/request ID 的原子
manifest：文本对应 `CF_UNICODETEXT`，文件对应 `CF_HDROP + Preferred DropEffect(COPY)`，图片对应
注册格式 `PNG` 和 `CF_DIBV5`。宿主在打开剪贴板前为全部格式分配
`GHND (GMEM_MOVEABLE | GMEM_ZEROINIT) HGLOBAL`，随后
`EmptyClipboard` 并按最有信息量的格式优先顺序用非空句柄调用 `SetClipboardData`；任一格式失败便
清空整个事务，未转移的句柄由宿主释放，成功转移的句柄归系统所有。该 eager 提交不依赖
`WM_RENDERFORMAT`、Qt 事件循环或拥有者进程继续存活。

提交 ACK 不是“函数没有抛异常”，而是关闭剪贴板后同时验证 sequence 已前进、owner 是本代宿主窗口、
`ClipSoon.InternalWrite` 的 request marker 一致、该类型所有格式可用（图片还要求系统可合成
`CF_DIB`）。同一 request ID 的重放是幂等的。SelectionSender 在 ACK 前不隐藏面板、不恢复目标、
不发送粘贴键；目标激活并经过配置延迟后，宿主再次校验同一 sequence/owner/marker/formats。若内容
被其他程序覆盖或宿主换代，只允许重新提交一次并再次校验，仍失败则取消自动粘贴。迟到、重复、
错误 request/kind/sequence 的响应只能被丢弃，不能触发第二次粘贴。

若采集外部剪贴板时 `GetClipboardData` 被延迟渲染永久阻塞，发送请求不会排在该调用之后：supervisor
终止旧宿主，在新 session 中用相同 request ID 重新准备并提交，最多尝试两代。进程级 10 秒原生操作
上限、请求级 22 秒 deadline 和粘贴前 5 秒 verify deadline 形成有界失败路径。标签构建的 Windows
冻结包 smoke 会真实发送文本、文件和图片，并在写入宿主退出后由独立 Win32 读取器验证 PNG、
CF_DIBV5 及系统合成的 CF_DIB 仍然存在。

窗口弹出前，macOS 记录前台进程 PID，Windows 记录前台窗口句柄。macOS 写入 pasteboard 后隐藏面板，
等待可配置延迟后恢复目标并发送 `Cmd+V`；Windows 按上述 ACK/verify 状态机发送 `Ctrl+V`。Windows
只在 `IsIconic` 确认目标已最小化时调用 `SW_RESTORE`，普通或最大化目标只置顶并设为前台，避免自动
粘贴改变用户的窗口尺寸状态。

Windows 的 Qt `activateWindow()` 不能保证把非活动应用带到前台，因此应用在 `show()` 后使用原生
窗口句柄请求 `SetForegroundWindow` 并验证结果。若普通请求失败，Qt GUI 线程会用
`GetWindowThreadProcessId + AttachThreadInput` 临时加入当前前台输入队列，在 `finally` 中解除附加，
重试后仍以 `GetForegroundWindow` 复核，成功才设置搜索框焦点。该兜底不向 helper 投递可能迟到并
重新显示已隐藏窗口的激活命令。面板可见期间以 35 ms 定时器
读取 `GetForegroundWindow` 和鼠标主键状态：一旦面板曾成为前台，后续前台变化即隐藏；若系统始终
拒绝激活，则以前台基线变化或面板外首次鼠标按下作为隐藏兜底。模态窗口、应用弹出菜单及关闭
“失焦隐藏”时只同步按键边沿而不隐藏。该策略不安装额外全局鼠标钩子，也不吞掉目标窗口点击。
Windows 禁用 `ClipPanel.changeEvent()` 的通用 35 ms 失焦隐藏，避免它早于原生激活重试把刚显示的
面板隐藏；延迟激活带本次显示 generation，旧回调不能影响后一次显示。快捷键再次触发时，仅当面板
确实是原生前台窗口才切换为隐藏；可见但未获前台时会重新激活，而不是把用户看不到的窗口隐藏。

搜索框的 Enter 门禁只跟踪该控件收到的 `QInputMethodEvent.preeditString()`。输入法候选窗或
虚拟键盘处于可见状态不代表正在组字，因此不能使用 `QInputMethod.isVisible()` 阻止发送；只有
确实存在 preedit 文本时，当前 Enter 才交给输入法完成候选确认。

Windows 主面板使用不透明的无边框顶层窗口和实色调色板，不设置 `WA_TranslucentBackground`，
从而完全绕过 Qt 的 `UpdateLayeredWindowIndirect` 分层窗口提交路径；同时不安装会扩大绘制脏区的
`QGraphicsDropShadowEffect`。macOS 继续使用透明窗口并保留原投影。QSS 使用 `pt` 字号，保证样式
和 delegate 继承的 `QFont.pointSize()` 始终为正。
macOS 快捷键服务同时检查 pynput 的 `running` 与线程 `is_alive()`，应用每 2 秒检查并按 15 秒退避
重建；Windows 由独立 supervisor 以心跳主动替换失效子进程。Python 未处理异常写入轮转日志，
主进程原生崩溃由 `faulthandler` 写入 `native-crash.log`；helper 错误通过协议回传主日志，避免三个
进程并发轮转同一个文件。

用户级开机自启动由 `LaunchAtLoginManager` 隔离：macOS 原子写入
`~/Library/LaunchAgents/com.clipsoon.app.plist`，Windows 写入
`HKCU\Software\Microsoft\Windows\CurrentVersion\Run` 的 `ClipSoon` 字符串值。PyInstaller 环境只记录
当前应用可执行文件；源码环境记录当前虚拟环境解释器和 `-m clipsoon`，Windows 优先使用同目录
`pythonw.exe` 避免登录时出现控制台。设置保存失败时保留原设置值并反馈；已开启时每次应用启动
都会重写启动项，使应用升级或移动后的可执行路径得到刷新。

## 6. 性能路径

应用启动后常驻并预构建窗口、内存搜索索引和列表 delegate。热键呼出只记录前台目标、计算
当前屏幕位置并显示已经存在的窗口，不查数据库。搜索字段已归一化，列表由 `QListView` 虚拟
绘制；图片 PNG 编码、数据库写入、列表缩略图和右侧大图解码均放入 Qt 线程池。选中路径只同步
更新当前索引、背景和元信息，图片先显示加载占位，后台返回缩放后的 `QImage` 后才在 GUI 线程
转换为 `QPixmap`，并按路径和目标尺寸缓存。日志记录冷启动、呼出和超过 20 ms 的搜索，用于
回归定位。

可选状态记忆只保存在当前进程内：面板真正隐藏前以单调时钟记录类型 Tab、搜索词、稳定条目 ID
集合和当前焦点 ID，并启动单次到期计时器。时限内重显时先恢复类型、搜索词和过滤结果，再按
ID 恢复选择；到期时主动丢弃整个快照、恢复“全部”、清空搜索并把隐藏面板重置到完整列表第一项。持久化配置仍沿用
`remember_selection` 键以兼容 0.9.1 用户设置，但该开关的产品语义已扩展为完整面板状态。

主面板位置以 `panel_x`、`panel_y` 两个可选逻辑坐标写入同一 JSON 设置。首次运行仍按当前鼠标
屏幕居中偏上显示；用户只能从非交互空白区域开始拖动，释放鼠标且位置确实改变后才由面板信号
触发一次设置落盘。后续显示优先选择保存坐标所在屏幕，并按该屏幕可用区域约束完整窗口；旧屏幕
不存在时回退到当前鼠标屏幕，避免窗口留在不可见区域。

开发验收统一通过仓库根目录的 `run.command` 或 `run.bat` 从当前源码启动。启动
脚本先终止本仓库的旧打包/源码实例，避免单实例锁让旧 `dist` 产物继续驻留并造成源码与界面
不一致。发布和最终包验收分别使用根目录的 `build_macos.command` 或 `build_windows.bat`；它们
转发到 `scripts/` 下的平台 PyInstaller 实现。

GitHub Actions 的 `.github/workflows/release.yml` 在推送 `v*` 标签时运行。Windows 任务固定使用 x64
托管 runner，macOS 任务固定使用 macOS 15 Apple Silicon runner；两者分别生成 ZIP 后作为独立
artifact 上传。发布任务只在两个平台构建均成功后运行，下载产物、生成 SHA-256 校验文件，
再使用最小的 `contents: write` 权限创建或更新对应的 GitHub Release。

界面主题由 `_style_sheet()` 提供主窗口与设置窗口的基础颜色和字体层级。主搜索框使用 16 pt，分类 Tab、
历史列表、文本预览和信息区标题/标签/值统一使用 13 pt；搜索图标为 30 px，搜索框高度根据搜索文字
的可见字形动态计算，使文字到边框顶部及底部的距离均约为文字高度的一半。列表 delegate 使用 44 px
行高和 30 × 30 px 缩略图容器，继承列表字号并只使用中等字重，避免搜索、分类、列表、详情之间出现
“大标题配小正文”的割裂感。弹出层不能只依赖父窗口继承。
macOS 会把 `QComboBox` 的下拉视图放在独立顶层窗口中，应用主题与系统主题不一致时可能继续使用
系统背景；因此两个设置下拉框会直接为弹出视图设置 Base/Text/Highlight 调色板及普通、悬停、
选中、禁用 QSS。历史右键菜单也接收 ClipSoon 当前主题值，而不是再次从系统调色板推断深浅；
系统托盘菜单保持操作系统原生外观。


操作系统可能拒绝跨进程激活；此时不能把“已复制”误报成“已粘贴”。应用保留剪贴板内容并
给出降级提示。
